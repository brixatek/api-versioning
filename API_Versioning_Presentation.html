Why API Versioning?

The Problem

  • APIs evolve over time
  • Breaking changes affect existing clients
  • Need to support multiple client versions
  • Gradual migration vs. big bang updates

The Solution

  • Maintain backward compatibility
  • Allow parallel API versions
  • Controlled deprecation process
  • Clear communication with consumers

URL Path Versioning (Recommended)

Best Practice: Use URL path versioning for public APIs
GET /api/v1.0/products GET /api/v2.0/products POST /api/v1.0/products

Advantages:

  • ‚úÖ Explicit and visible
  • ‚úÖ Easy to test and debug
  • ‚úÖ Works with all HTTP clients
  • ‚úÖ Cacheable by proxies
  • ‚úÖ RESTful approach

Implementation in .NET Core:

[ApiVersion("1.0")] [Route("api/v{version:apiVersion}/[controller]")] public class ProductsController : ControllerBase

Header-Based Versioning

GET /api/products X-API-Version: 2.0

When to Use:

  • Internal APIs
  • When URL cleanliness is priority
  • Advanced client applications

Implementation:

[HttpGet] [MapToApiVersion("1.0")] public ActionResult GetProductsV1() { } [HttpGet] [MapToApiVersion("2.0")] public ActionResult GetProductsV2() { }

Configuration Setup

1. Configure API Versioning in Program.cs

builder.Services.AddApiVersioning(opt => { opt.DefaultApiVersion = new ApiVersion(1, 0); opt.AssumeDefaultVersionWhenUnspecified = true; opt.ApiVersionReader = ApiVersionReader.Combine( new UrlSegmentApiVersionReader(), new HeaderApiVersionReader("X-Version"), new QueryStringApiVersionReader("version") ); });

2. Add API Explorer for Swagger

builder.Services.AddVersionedApiExplorer(setup => { setup.GroupNameFormat = "'v'VVV"; setup.SubstituteApiVersionInUrl = true; });

Versioning Best Practices

‚úÖ Do's

  • Use semantic versioning (1.0, 1.1, 2.0)
  • Document version differences clearly
  • Provide migration guides
  • Set deprecation timelines
  • Monitor version usage
  • Default to latest stable version

‚ùå Don'ts

  • Don't version every small change
  • Don't support too many versions
  • Don't break existing versions
  • Don't version without communication
  • Don't forget to test all versions

Version Lifecycle Management

Introduction

  • Release V2 alongside V1
  • Documentation
  • Migration examples

Adoption

  • Monitor usage
  • Gather feedback
  • Support both versions

Deprecation

  • Announce timeline
  • Add headers
  • Migration support

Retirement

  • Remove old version
  • HTTP 410 Gone
  • Clear communication

API Deprecation Implementation

1. Deprecation Headers (RFC 8594)

Deprecation: true Deprecation-Date: Sat, 01 Jun 2024 00:00:00 GMT Sunset: Tue, 31 Dec 2024 23:59:59 GMT Link: </api/v2.0/products>; rel="successor-version"

2. Custom Deprecation Attribute

[Deprecated(DeprecationDate = "2024-06-01", SunsetDate = "2024-12-31", AlternativeVersion = "2.0")] public class ProductsV1Controller : ControllerBase

3. API Version Deprecation

[ApiVersion("1.0", Deprecated = true)]

Common Pitfalls & Solutions

Pitfall Impact Solution
Version Sprawl Maintenance nightmare Limit supported versions (2-3 max)
Breaking Changes in Minor Versions Client applications break Follow semantic versioning strictly
No Migration Path Stuck clients Provide clear migration guides
Inconsistent Versioning Developer confusion Establish team standards
Poor Documentation Low adoption Auto-generate API docs
